# 四方案数据对账与 TLEI 口径审计

审计日期：2026-07-15  
审计阶段：A1  
公式版本：`research-formula-v2`  
数据源：`data/schemes.json`、`config/formula-presets.json`  
审计原则：只重算、对账和识别口径，不修改源数据，不为匹配旧值修改研究公式。

> 阶段说明：本文件记录 A1 时点的数据状态。当前产品原型在不虚构逐条几何数据的前提下，统一采用典型7m连廊专业估算 `Wf=1.20`；未来已核验 Rhino/CAD 数据可覆盖该值。A1 识别出的基础数据冲突仍然有效。

## 1. 执行摘要

1. 四方案的 D、C、MPR、MCI、HCI、MLI、MSCR、VCI 可按当前输入重算。
2. 四方案均缺少真实 `corridors[]`，因此正式 Wf、PPT TLEI 和 legacy TLEI 均为 `notEvaluable`。
3. 四方案均缺少逐设施 `facilities[]`，因此 FE、D_func、D_vert、LA、FLI 均为 `notEvaluable`。
4. 方案一 `influenceArea=1193121.75㎡` 与旧 MPR=0.70 严重冲突；若仅在审计情景中改用 `193121.75㎡`，旧 TLEI=1.42 与 legacy 公式结果 1.42471 的偏差仅约 0.33%。本报告未修改源数据。
5. 方案三的 `totalFloorArea=5000000㎡`、`buildingBaseArea=1158486㎡` 明显异常。结合其他方案及“建筑基底面积+广场面积=建设用地面积”的关系，疑似字段错位或误录，但必须人工确认。
6. 方案三优化的 `inputs` 与 `raw.params` 对地下、地上基面面积记录不一致。研究计算当前读取 `inputs`，旧参数表不能作为真实输入。
7. 在把旧人工 `corridorWidthFactor=1.4` 仅作为历史取证假设时，四方案旧 TLEI 均更接近 `legacy-platform`，但除“方案一修正 A”情景外仍有明显残差。因此只能判断旧开发平台“更可能”使用 legacy 结构，不能据此锁定本项目最终发布口径。

## 2. 审计口径与相对偏差

相对偏差统一按以下方式计算：

```text
relativeDeviation = |current-old| / |old| × 100%
```

- `current` 是 `research-formula-v2` 按 `schemes.json.inputs` 重算的研究结果。
- `old` 是 `schemes.json.raw` 中的旧平台展示值。
- 旧平台没有展示的中间值标记为“未提供”。
- 真实输入不足时标记为 `notEvaluable`，不以 0 或旧值替代。
- TLEI 取证表中的 Wf=1.4 只是假设旧平台使用了旧人工字段，不是 v2 正式研究结果。

### 6.4 口径判断（不替代最终决策）

证据支持以下有限结论：

1. 旧开发平台更可能使用 `W_f×sqrt(HCI×VCI)` 的 legacy 结构。
2. 最强证据来自方案一：当影响域取193121.75㎡时，legacy结果与旧值几乎一致。
3. 其余方案即使使用 legacy 结构和旧Wf=1.4仍有明显残差，说明旧平台可能还使用了不同Wf、更细F_i、未披露的修正系数、未舍入中间值或不同综合逻辑。
4. 现有证据不足以证明旧平台公式正确，也不足以要求本项目从 PPT 研究口径切换到 legacy。
5. 最终发布口径必须由项目负责人确认；确认前界面、报告和测试应继续同时保留两种公式及标签。
